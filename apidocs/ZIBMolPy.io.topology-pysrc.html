 47  # Comments on optional fields in the atomtypes section: 
 48  #  
 49  # The force field format is getting a bit old. For OPLS-AA we needed 
 50  # to add a special bonded atomtype, and for Gerrit Groenhofs QM/MM stuff 
 51  # we also needed the atomic numbers. 
 52  # To avoid making all old or user-generated force fields unusable we 
 53  # have introduced both these quantities as optional columns, and do some 
 54  # acrobatics to check whether they are present or not. 
 55  # This will all look much nicer when we switch to XML... sigh. 
 56  #  
 57  # Field 0 (mandatory) is the nonbonded type name. (string) 
 58  # Field 1 (optional)  is the bonded type (string) 
 59  # Field 2 (optional)  is the atomic number (int) 
 60  # Field 3 (mandatory) is the mass (numerical) 
 61  # Field 4 (mandatory) is the charge (numerical) 
 62  # Field 5 (mandatory) is the particle type (single character) 
 63  # This is followed by a number of nonbonded parameters. 
 64  #  
 65  # The safest way to identify the format is the particle type field. 
 66  #  
 67  # So, here is what we do: 
 68  #  
 69  # A. Read in the first six fields as strings 
 70  # B. If field 3 (starting from 0) is a single char, we have neither 
 71  #    bonded_type or atomic numbers. 
 72  # C. If field 5 is a single char we have both. 
 73  # D. If field 4 is a single char we check field 1. If this begins with 
 74  #    an alphabetical character we have bonded types, otherwise atomic numbers. 
 75  #    atomtypes have more columns but e.g. the diala_quick has a strange 3rd column
